# Участие в разработке

Спасибо за интерес к **zapret-ui**! Принимаются и баг-репорты, и pull request'ы.
Этот документ описывает, как собрать проект, как устроена кодовая база и каких
правил придерживаться при отправке изменений.

> Участвуя в проекте, вы соглашаетесь соблюдать
> [Кодекс поведения](CODE_OF_CONDUCT.md).

## С чего начать

- **Нашли ошибку?** Откройте [issue](https://github.com/MelDxKviel/zapret-ui/issues)
  по шаблону «Сообщить об ошибке».
- **Есть идея?** Заведите issue по шаблону «Предложить улучшение» — лучше
  обсудить крупную правку до того, как писать код.
- **Не помогает ни одна стратегия у вашего провайдера?** Это почти всегда вопрос
  к ядру обхода, а не к оболочке — сообщайте в апстрим
  [Flowseal/zapret-discord-youtube](https://github.com/Flowseal/zapret-discord-youtube).

## Окружение для сборки

- **Только Windows 10/11 x64.** Большая часть кода использует Win32 FFI и
  `windows-service`, поэтому собрать или запустить проект на других ОС нельзя.
- **Rust 1.95.0** — версия закреплена в [`rust-toolchain.toml`](rust-toolchain.toml),
  rustup подтянет её автоматически. Нужен MSVC-таргет (`x86_64-pc-windows-msvc`) и
  установленные Build Tools for Visual Studio.

```powershell
cargo build --release        # → target\release\zapret-ui.exe (один файл, без DLL)
cargo test                   # все тесты
cargo test <имя_теста>       # один тест по имени
cargo clippy                 # линтер (CI требует, чтобы было чисто)
cargo fmt                    # форматирование перед коммитом
```

### Быстрая итерация по UI

```powershell
cargo run --example ui_only
```

Этот пример поднимает интерфейс Slint с **мок-бэкендами** — никакого реального
скачивания, процессов или службы, без прав администратора и без UAC. Самый
быстрый способ дорабатывать UI.

> ⚠️ Релизные сборки несут манифест `requireAdministrator` и всегда стартуют с
> запросом UAC. Отладочные сборки (`cargo run`, `cargo run --example ui_only`,
> тесты) остаются `asInvoker` и работают без повышения прав. Часть тестов службы
> по-настоящему обращается к SCM только при запуске из-под администратора —
> иначе они проверяют путь `NeedsElevation`.

## Архитектура в двух словах

Проект построен по принципу **портов и адаптеров**. Подробное описание — в
[`CLAUDE.md`](CLAUDE.md); коротко:

- `src/ports.rs` — трейты (`Installer`, `Runner`, `ServiceCtl`, `StrategyCatalog`,
  `StrategyTester`, `Maintenance`).
- `src/contracts.rs` — общие типы данных (`Strategy`, `RuntimeStatus`,
  `BackendCmd`, `UiEvent`).
- `src/zapret/` — конкретные адаптеры (парсер `.bat`, установщик, раннер, служба,
  тестер и т.д.).
- `src/app.rs` — оркестратор; владеет `Arc<dyn Trait>` и не зависит от
  конкретных адаптеров (поэтому `examples/ui_only.rs` подменяет их моками).
- `ui/` — интерфейс на Slint, компилируется через `build.rs`.

Обмен сообщениями между UI и бэкендом идёт по двум каналам: команды
(`BackendCmd`) уходят в mpsc-канал, события (`UiEvent`) приходят обратно по
broadcast-каналу. **Не трогайте свойства Slint напрямую из фоновых задач** —
только через `UiEvent` + слушатель или `slint::invoke_from_event_loop`.

## Правила, которые легко нарушить

- **Контракт UI ↔ Rust.** Имена `callback`/`in-out property` в
  `ui/main_window.slint` — ручной контракт сразу с `src/app.rs` **и**
  `examples/ui_only.rs`. Добавили или переименовали callback/property — обновите
  оба Rust-файла, иначе сборка упадёт.
- **i18n.** Каждая видимая пользователю строка — это `I18n.t(I18n.lang, "ключ")`.
  Ключи лежат в плоских JSON-каталогах `src/locales/ru.json` и
  `src/locales/en.json`. Юнит-тест проверяет, что наборы ключей совпадают —
  **добавляете строку, добавляйте её в оба файла**.
- **Стратегии не хардкодятся** — они находятся в рантайме сканированием `.bat` в
  каталоге установки (`LocalStrategyCatalog`). Не возвращайте захардкоженный
  список.
- **Никаких обращений к `api.github.com`** — он заблокирован DPI у целевых
  провайдеров. Версия берётся с `raw.githubusercontent.com`, архивы — с
  `codeload.github.com`/`releases/download`.

## Pull request'ы

1. Создайте ветку от `main`.
2. Делайте сфокусированные коммиты с понятными сообщениями (повелительное
   наклонение: «Add…», «Fix…»). Крупную несвязанную правку лучше разнести на
   отдельные PR.
3. Перед отправкой убедитесь, что проходят:
   ```powershell
   cargo fmt
   cargo clippy
   cargo test
   ```
4. Если правка меняет поведение, опишите в PR, **как вы её проверили** (особенно
   то, что не покрыто тестами — реальный запуск/служба/UAC).
5. Заполните шаблон PR и свяжите PR с issue (`Closes #123`), если он есть.

CI на `windows-2022` гоняет clippy + тесты на каждый push и PR — зелёный CI
обязателен для слияния. Релизы выпускаются тегом (`vX.Y.Z`), подробности — в
[`docs/RELEASING.md`](docs/RELEASING.md).
